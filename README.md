Проект: база данных "Салон красоты"
О проекте
Этот проект представляет собой реляционную базу данных, разработанную для моделирования работы салона красоты. Система отслеживает:

Клиентов и их записи на услуги
Мастеров (парикмахеров, косметологов, массажистов и др.)
Перечень предоставляемых услуг с описанием, ценой и длительностью
Расписание записей (дата, время, статус)
Продажу косметики и уходовых средств
Отзывы клиентов с оценками и комментариями
База данных может служить основой для реального CRM-системы салона красоты или использоваться в учебных целях для изучения проектирования реляционных моделей, нормализации данных и работы с внешними ключами.

Сущности (таблицы)
clients: Клиенты салона — ФИО, телефон, email, дата регистрации
masteri: Мастера — ФИО, должность, зарплата, дата приёма на работу
services: Услуги — название, описание, цена, длительность в минутах
zapisi: Записи на приём — дата и время, статус, ссылки на клиента, мастера и услугу
kosmetika: Косметика — название, бренд, цена, остаток на складе
sales: Продажи — дата и время покупки, клиент, товар, количество, сумма
feedback: Отзывы — текст, оценка (1–5), дата, привязка к клиенту и конкретной записи

Связи между сущностями
Схема связей построена на принципах реляционной целостности и включает следующие зависимости:

clients → zapisi
Один клиент может иметь множество записей
(zapisi.client_id → clients.client_id)
masteri → zapisi
Один мастер может вести множество записей
(zapisi.rabotnik_id → masteri.rabotnik_id)
services → zapisi
Одна услуга может быть назначена в множестве записей
(zapisi.service_id → services.service_id)
clients → sales
Один клиент может совершить несколько покупок
(sales.client_id → clients.client_id)
kosmetika → sales
Один товар может быть продан многократно
(sales.product_id → kosmetika.product_id)
clients → feedback
Один клиент может оставить несколько отзывов
(feedback.client_id → clients.client_id)
zapisi → feedback
Одна запись порождает не более одного отзыва (связь «один к одному»)
(feedback.zapisi_id → zapisi.zapisi_id, поле zapisi_id в feedback — уникальное)
Все связи реализованы через внешние ключи. Структура соответствует третьей нормальной форме (3НФ): отсутствует избыточность данных, все неключевые атрибуты зависят только от первичного ключа, и нет транзитивных зависимостей.
